# 애셋 통합 프로토콜 (외부 팩·에셋 도입 시 상시 절차)

> **목적**: "팩의 설계 규약을 잘못 가정한 채 저작"해서 생기는 반복 왕복(재작업)을 막는다. 무기뿐 아니라 **모든 애셋 타입**(캐릭터·VFX·오디오·애니·환경·머티리얼)에 적용.
>
> **근거 원칙**: 프로젝트 CLAUDE.md의 **ENGINE/ASSET SOURCE FIRST**(원본에서 실제 사용례 먼저 조사)와 **DESIGN-FIRST**(기존 패턴 전수조사)를 **외부 애셋 팩에도 적용**한다. 추론·상속된 설정을 답습하지 말고, **팩의 규약을 저작 전에 역설계**하고, **조립된 결과를 검증**하고, **데이터 참조를 자동 검증**한다.

## 이 문서가 만들어진 계기 (실제 실패 사례)
애니 패스에서 발생한 왕복들은 전부 같은 뿌리였다 — 팩 규약을 **선제적으로** 파악하지 않고 버그가 터질 때마다 **반응적으로** 하나씩 발견:
- Rifle을 정적 프리뷰(MAK12)만 보고 저작 → 실제 의도는 AG14W (**의도 매핑 미확인**)
- 총구 소켓을 리시버에 추측 배치 → 실제 팩은 **배럴/포어스톡 파트에 `SOCKET_Muzzle`** 심어둠 (**파트 소켓 미조사**)
- 재장전 시 새 탄창 안 보임 → 팩은 **2탄창 스왑**(`SOCKET_Magazine`/`_Reserve`) (**리로드 본트랙 미조사**)
- `MuzzleSocket="SOCKET Muzzle"`(공백) 오타 → 조회 실패 (**데이터 참조 미검증**)

## 4단계 프로토콜

### 1. DISCOVER — 저작 전 규약 역설계 (절대 생략 금지)
팩에서 **첫 애셋을 저작하기 전에** 1회, 팩 전체 규약을 조사해 짧은 "팩 통합 스펙" 노트로 남긴다(같은 팩의 모든 애셋에서 재사용).
- **구조/네이밍**: 폴더 구조, 프리픽스 규약(SK_/SM_/A_/SOCKET_ 등).
- **메시 소켓**: **주 메시뿐 아니라 모든 모듈 파트 메시**의 소켓을 전수 열거. ⚠️ 소켓 vs 본 구분(팩이 본을 `SOCKET_*`로 명명하는 경우 많음 — 런타임 attach는 둘 다 됨).
- **스켈/애니**: 스켈레톤 공유 여부, **additive vs non-additive**(슬롯 몽타주엔 non-additive), 핵심 애니의 **본 트랙 실측**(예: 리로드의 탄창 스왑), 노티파이.
- **머티리얼**: blend mode(opaque/translucent), 머티리얼 슬롯(무슨 지오메트리가 메시에 있고 무엇이 모듈 파트인지).
- **VFX/오디오**: 분사 축·방향 규약, attach 기대치.
- **산출물**: 팩별 "통합 스펙" 노트(무기 예 = [WeaponPack_Integration.md](WeaponPack_Integration.md)).

### 2. MAP INTENT — 사용자와 매핑 확정
- **어떤 팩 아이템 = 어떤 게임 엔티티**인지 확정. **상속·플레이스홀더·프리뷰 설정을 신뢰하지 말 것**(Rifle 프리뷰=MAK12였으나 의도=AG14W). 애매하면 저작 **전에** 사용자에게 30초 확인 — 이 확인 하나가 전체 재작업을 막는다.

### 3. AUTHOR — 규약대로, 올바른 메커니즘
- 팩이 의도한 **메커니즘**을 그대로(파트 소켓, 2탄창 스왑 등). "되는 대로 제일 쉬운 길"이 아니라 팩 설계에 맞춤.
- 에셋 경로 **C++ 하드코딩 금지** — 전부 DataAsset soft ref.

### 4. VERIFY — "완료" 선언 전 (기능 + 조립 비주얼 + 데이터)
- **기능**: 컴파일, 스켈 매칭, 슬롯명 일치.
- **조립 비주얼**: 조립된 애셋을 **오프라인 렌더/실측으로 내가 먼저 확인**(임시 스폰→측정/스크린샷→**삭제**; 레벨에 프리뷰 액터 남기지 말 것). 컴파일·기능만 보고 "완료" 하면 조립 결함(총구 위치/부품/탄창)을 사용자가 먼저 발견하게 된다.
- **데이터 검증**: `IsDataValid` + **참조 이름 존재 검사**(소켓·본·파라미터). 헤드리스는 `is_object_valid`/커맨드릿(쿠킹·CI).
- **파이썬 저작 파생 데이터 자산은 COLD 검증**: BlendSpace 등 런타임 파생 데이터를 갖는 자산을 파이썬으로 만들면 내부 데이터가 미빌드돼 **런타임에만** 깨질 수 있다(에디터 프리뷰는 정상). 저작 후 **수동 리컴파일 없이 새 PIE**로 검증하고, 필요 시 강제 리빌드(open_editor)+참조 자산 재컴파일 또는 에디터 직접 생성. 상세 = `Docs/WeaponPack_Integration.md`.

## 애셋 타입별 빠른 체크리스트

**스켈레탈 메시 / 무기**: 모든 파트 메시 소켓 열거 · 총구=파트 소켓인지 · 리로드/장전 본트랙(멀티 파트 스왑?) · additive 여부 · 스켈 공유. → 상세 [WeaponPack_Integration.md](WeaponPack_Integration.md).

**VFX (Cascade/Niagara)**: **분사 축**(대개 로컬 +X)이 소켓 방향과 맞는지 → 안 맞으면 소켓 회전 보정(내부 분사축이 보호돼 못 읽으면 조립 후 검증). attach 컴포넌트가 맞는지(파트 vs 리시버).

**오디오**: attenuation·공간화, attach 대상, 3P 가청성.

**애니메이션**: additive/스켈 매칭/슬롯명, 루트모션, 노티파이(팩 노티파이를 핸들 안 하면 스왑/토글 로직 누락).

**환경/머티리얼**: blend mode, LOD, 콜리전, Nanite/라이트맵.

## 상시 자동 검사 (재발 방지 코드)
- **DA `IsDataValid`가 참조 소켓 존재를 검증**한다 — 무기 DA는 구현됨(`UFPSRWeaponDataAsset::IsDataValid`: `MuzzleSocket`·`WeaponParts1P` 소켓이 실제 메시/파트에 있는지, 공백 오타 등 차단). **새 애셋 타입의 DA를 만들 때 같은 패턴으로 참조 이름 검증을 추가**한다.
- 참조 이름(소켓·본·머티리얼 파라미터·데이터테이블 로우 키 등)을 문자열로 들고 있는 DA는 **그 이름이 대상에 실제 존재하는지 IsDataValid에서 검사**하는 것을 표준으로 한다.
- IsDataValid는 WITH_EDITOR 전용이라 빌드/스모크가 무효 DA를 못 잡음 → 쿠킹/CI에서는 헤드리스 검증 병행([[vibeue-render-target-gpu-hazard]]).
