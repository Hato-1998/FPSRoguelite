# PlanConsultLoop — 플랜 특화 컨설트 프로토콜 (plan-first 델타)

> **이 문서는 ConsultLoop의 plan-first 델타다.** 공유 토론 골격(역할 분리, read-only 원칙, 라운드 호출법, `Docs/Review/` 산출·`_raw/` 보관, 자문 전용, SSOT/PROGRESS 선독)은 [`Docs/ConsultLoop.md`](ConsultLoop.md)가 소유한다. **같은 규칙이 양쪽에 있으면 여기서 지우고 ConsultLoop을 링크하라.** 여기엔 `/consult`와 다른 것만 적는다.
>
> Shared Contract: ConsultLoop v1 · Plan Delta: PlanConsult v1

---

## §0 목적 · `/consult`와의 차이
- `/consult` = 주제 자문(리뷰). **`/plan-consult` = 사용자가 낸 작업(task)을 먼저 실행 가능한 플랜 초안으로 바꾼 뒤, 그 플랜을 Codex로 적대 수렴시켜 결정지원 리포트를 낸다.**
- 3대 차별 불변식: **(a) plan-first** — 리뷰가 아니라 '내가 쓴 플랜'을 공격받는다. **(b) 강화 출력** — 쟁점의 '발생 경위' + 사용자 결정의 '선택 기준'을 반드시 낸다. **(c) delta-gated 수렴** — 라운드 '수'가 아니라 '새 정보 생산 여부'가 브레이크다.

## §1 Intake gate — LIGHT / FULL / NO-GO
목적: 절차 무게를 작업 위험에 맞춘다(가벼운 도구로 생존). **문턱은 "큰 작업"이 아니라 "잘못된 계획의 되돌리기 비용이 큰 작업".**
- **NO-GO** — 요구가 불명확(먼저 사용자에게 1회 질문), 또는 HIGH_RISK 승인·SSOT 갱신·사용자 결정이 선행돼야 함. → 루프를 돌리지 말고 그 선행을 먼저 처리.
- **LIGHT (기본)** — 작지만 사소하진 않은 작업. 10줄 이내 초안 + Codex 1라운드(질문 최대 2개) + 라운드 변화 3줄 요약 + `Exit Check` 1줄. 산출은 작업 플랜 또는 `PROGRESS.md` 하단. 30–45분 상한(넘으면 FULL 승격 또는 종료).
- **FULL** — 하드트리거 해당 또는 결정축 다수. 전체 절차 + `Docs/Review/` 저장.
- **하드트리거(내용 무관 FULL)**: 서버권위/복제/RPC/Push Model/맵 이동/SaveGame 관여, 대량 적 성능경로(스폰·AI tick·pathing·대량 VFX) 변경, 데이터 마이그레이션, phase 경계를 흔드는 구조 변경, 결정축 3개 이상.
- (참고: 정량 intake 점수표는 v1 미채택 — gate만 쓰고 본 루프를 안 쓰게 만드는 대표적 과설계. 실제 필요가 보이면 후속에서 추가. §7.)

## §2 plan_type 판정 + 렌즈 R0 고정
R0에서 `plan_type`을 판정하고 **판정 근거(읽은 파일·task 키워드·변경 대상·실패 비용)를 리포트에 남긴다.** 판정 후 Codex 적대 페르소나를 **끝까지 고정한다(라운드 중 바꾸지 않음):**
- `backend/system` → **제2 아키텍트/레드팀**(계약·확장성·검증성·운영 비용·되돌리기·기존 파이프라인 충돌).
- `player-facing/content` → ConsultLoop의 **클라이언트/콘텐츠 렌즈**(손맛·입력·UI·연출·콘텐츠 실현성).
- `tooling/workflow` → **툴 사용성·실패모드·중복·유지보수 레드팀**.
- `mixed` → primary 렌즈 고정 + 종료 전 secondary 축 1회 강제(라운드마다 스왑 금지).

적응은 '렌즈 선택'까지만이다 — 다중 렌즈 분기표는 DEFER(§7). 핵심은 "선택된 반대자 페르소나를 끝까지 유지"하는 것.

## §3 절차 (6 스텝)
1. Intake 판정(§1).
2. 최소 컨텍스트 로드(`Game.md`·`PROGRESS.md`·해당 SSOT) + `plan_type`/렌즈 고정(§2).
3. 초안 플랜 작성(목표·제약·proposed plan·사용자 결정지점·검증법). 코드/에셋 변경 없음.
4. Codex 적대 리뷰(ConsultLoop §4 호출법 재사용). 질문은 좁힌다: "이 플랜이 과설계인가 / 실행을 막는 결함이 있나 / 사용자가 결정해야 할 쟁점은?".
5. 수렴 — Codex 지적을 `수용 / 기각 / 사용자 결정 필요`로 분류. 라운드 정책(§4)에 따라 반복·종료.
6. 결과 리포트(§5).

## §4 라운드 정책 (delta-gated core)
**핵심 원칙: 쟁점 자체는 성과가 아니다. 쟁점이 플랜 변경(`PLAN_DELTA`)·명시적 리스크 수용(`RISK_ACCEPTANCE`)·반증가능 검증훅(`FALSIFIABLE_CHECK`) 중 하나로 변환될 때만 성과다.** 아무것도 안 나오면 그 라운드는 `NO_DELTA`.
- 라운드는 새 delta가 나오는 한 계속한다. 정지 = "요구된 축 소진 후 다음 최선 축에서도 `NO_DELTA`". 2연속 `NO_DELTA`면 종료(또는 사용자 승인).
- **조기·얕은 수렴 방지**: 2라운드 이하로 합의처럼 보이면 종료 금지 → **Divergence Round**로 아래 축 중 미사용 2~3개를 재공격한다. 강제 문구: "이 라운드는 기존 합의 재진술 금지. 선택한 축 이름 + 새로 발견한 반례만."
  - **재공격 고정 축**: 전제 뒤집기 / 실패모드 / 비용·유지보수 / 기존 도구 충돌 / 검증 불가능성 / 역행(rollback) / 스코프 폭주 / 보일러플레이트화.

**정책 선택 (사용자 결정 항목 — rubric):**
1. `Bounded` — 최대 2라운드, `NO_DELTA`면 종료. 기준: 반복적 작업·저위험 결정·되돌리기 쉬움·이미 검증 루프가 강함.
2. `Delta-Gated` — 새 delta가 나오는 동안 계속, `NO_DELTA`면 종료. 기준: 일반적인 설계/툴링 결정, 논점은 있으나 축이 제한적.
3. `Deep Delta-Gated` **(이 프로젝트 기본값)** — 서로 다른 divergence 축 최소 N개 소진 후 delta-gated 종료. 기준: 되돌리기 비용 큼·장기 영향·조기 합의 의심·실패 모드 다양. 반복·저위험 작업은 Intake가 LIGHT로 자동 강등하거나 명시 시 Delta-Gated/Bounded로 낮춘다.
4. `Fixed-Minimum Audit` — 최소 N라운드 강제(각 라운드 서로 다른 축 의무 + `NO_DELTA`도 리포트에 '무성과'로 표기 + 같은 주장의 재표현은 delta 불인정 + N 이후 delta-gated로 전환) 후 종료. 기준: 관점 커버리지 자체가 요구되거나, 검토 기록이 의사결정 증거로 필요.

선택 rubric(취향이 아니라 근거로 고를 것): 되돌리기 비용↑ · 불확실성/미검증 가정↑ · 결정 영향 범위↑ → **깊게**. 반복 가능·저위험·검증 쉬움 → **상한형**.

## §5 출력 리포트 (5 섹션)
FULL은 `Docs/Review/<yyyyMMdd>-plan-<슬러그>.md`에 저장. LIGHT는 §1대로 축소.
1. **Intake** — `Mode: LIGHT/FULL/NO-GO` + 이유 1줄 + `plan_type`/렌즈 + 라운드 정책.
2. **결과 플랜 (수렴)** — 실행 가능한 최종 플랜 + 범위 밖으로 뺀 것.
3. **수렴 로그** — 초안에서 무엇이 바뀌었나(라운드별 변화 요약) + Codex 지적 중 수용/기각/보류.
4. **미해결 쟁점 · 사용자 결정 필요** — 항목마다:
   - 쟁점: **발생 경위(몇 R에서 어떤 반론 때문에 나왔나)** + 백엔드 이유 + Codex 이유 + 현재 상태.
   - 사용자 결정: **왜 사람이 정해야 하나** + 선택지 + **선택 기준(정량·정성 rubric)**.
5. **검증 상태** — 확인됨 / 추정 / 검증 필요. 증거등급(SSOT 직접 인용 · 코드/소스 관찰 · 선례 · 추론 · 추측) 표기. 자문 전용이라 빌드/테스트로 미검증임을 명시. (FULL·고위험 플랜은 반증가능 예측 1~2줄 추가 권장: "이 플랜이 맞다면 …, 틀렸다면 첫 구현 체크포인트에서 … 신호".)

## §6 가드레일
1. **자문 전용** — 토론·플랜은 코드/에셋을 바꾸지 않는다. 채택된 설계는 별도 승인 단계에서 해당 `Docs/SSOT/` 파일을 먼저 갱신 후 구현.
2. **명명 정직** — Codex 미가용이면 산출물 제목·본문에 "Codex 미검증 초안"으로 격하한다. "수렴 리포트"라 부르지 않는다. 첫 호출 전 preflight로 호출 가능성 확인.
3. **`/pm` 이중 플래닝 방지** — 결과 = decision-support 리포트. 실행 백로그/작업 프롬프트 편성은 `/pm`·`Docs/TaskPrompts_Master.md`의 몫. 액션 아이템은 "PM 인입 후보"로만 기록.
4. **hard stop budget** — LIGHT=1라운드, FULL은 라운드 정책 상한을 따른다. 새 쟁점 없이 회의체화 금지.
5. **read-only** — Codex는 read-only 샌드박스로 호출(ConsultLoop §6-5).

## §7 v1 범위 밖 (DEFER — 은닉이 아니라 명시)
후속에서 실제 필요가 보이면 추가: 정량 intake 점수표 · Divergence 정량 지표 · Round Delta 태그 원장(정식 스키마) · Falsifiable 상시 필드 · telemetry frontmatter · drift-check 스크립트(`Scripts/check-consult-doc-drift.ps1`) · 다중 렌즈 분기표 · 2주/3건 trial 롤아웃 및 진입로 차단식 롤백. (근거: 1인 HotL 도구의 과설계 방지 — R4 티어링 결론.)
