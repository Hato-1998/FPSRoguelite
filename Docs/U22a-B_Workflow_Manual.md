## §1 기본 루프 — 모든 주제에 똑같이 적용

한 주제(도로, 건물, 광장…)마다 아래 6단계를 **순서대로** 밟는다. 단계를 건너뛰지 않는다.

```
 ① 주제 선언  →  ② 사용자 방향 제시  →  ③ Claude 에셋 실측  →
 ④ Claude 구체안 제시  →  ⑤ 토론 · 확정  →  ⑥ 생성 · 육안 확인
                                              ↑            ↓
                                              └─ 수정 ─────┘  (만족하면 다음 주제)
```

| 단계 | 누가 | 하는 일 | 산출물 |
|---|---|---|---|
| ① 주제 선언 | Claude | "이번엔 도로망을 정합니다" — 지금 무슨 주제인지 명시 | 한 줄 |
| ② 방향 제시 | **사용자** | 원하는 큰 그림을 말로. 수치 몰라도 됨 ("좁은 골목 많고 대로 하나") | 자유 형식 |
| ③ **에셋 실측** | Claude | 그걸 만들 에셋이 **실제로 있는지** 조사·측정 | **실측 카드**(§2) |
| ④ 구체안 제시 | Claude | 수치·개수·규칙으로 옮기고, 선택지 2~3개 + 추천 | **구체안 카드**(§3) |
| ⑤ 토론·확정 | 양쪽 | 되묻고 고침. 사용자가 **"확정"** 이라고 하면 종료 | §10에 기록 |
| ⑥ 생성·확인 | Claude → 사용자 | MCP Python으로 생성 → 사용자가 눈으로 확인 | 맵 변경 + 보고 |

### 루프 규칙
1. **③을 건너뛰고 ④로 가지 않는다.** 에셋이 있는지 모르는 채 수치부터 정하면 재작업이 난다.
2. **한 번에 한 주제.** 도로 확정 전에 건물 이야기로 넘어가지 않는다.
3. **사용자가 "확정"이라 말하기 전에는 생성하지 않는다.** (⑥은 승인 후)
4. ⑥에서 마음에 안 들면 **⑤로 되돌아간다.** 지우고 다시 만드는 건 §6 태그 규약으로 항상 가능하다.
5. Claude가 확신 없으면 **추측하지 말고 실측하거나 물어본다.**

## §2 ③단계 산출물 — 「에셋 실측 카드」

사용자가 원한 것 하나하나에 대해 아래 표를 낸다. **모든 크기는 실제로 메시를 열어 잰 값**이다(추정 금지).

```
■ 요청: "왕복 4차선 대로 + 인도"

| 원하는 것 | 후보 에셋 | 실측 크기(cm) | 개수 | 판정 |
|---|---|---|---|---|
| 차도 바닥 | (없음)                        | -            | 0 | 🔴 없음 |
| 인도      | SM_Bld_Base_Floor_01          | 250×250×?    | 1 | 🟡 조합 |
| 가로등    | SM_Prop_Street_Light_01~13    | 높이 ?       | 13| 🟢 있음 |
```

**판정 4종**
| 기호 | 뜻 | 다음 행동 |
|---|---|---|
| 🟢 있음 | 그대로 쓸 수 있음 | 바로 ④로 |
| 🟡 조합 | 팩 안의 다른 조각을 붙이면 됨 | 조합 방법까지 ④에 포함 |
| 🔵 타팩 | 이 팩엔 없지만 다른 보유 팩에 있음 | 룩이 섞이는지 검토 후 ④ |
| 🔴 없음 | 어디에도 없음 | **대안 사다리** 적용 ↓ |

**🔴일 때 대안 사다리** — 위에서부터 검토하고, 어디까지 내려갔는지 사용자에게 밝힌다.
1. 같은 팩의 다른 조각으로 대체 (룩 일관성 최고)
2. 다른 보유 팩에서 가져오기 (룩이 섞이는 정도를 반드시 보고)
3. **기본 도형 + 팩 머티리얼** (예: 도로 = 평면 메시에 팩의 아스팔트 머티리얼) ← 가장 흔한 착지점
4. 그 요소를 **설계에서 뺀다** (예: 차도를 안 만들고 보행자 전용 구역으로)
5. 새 에셋 구매/제작 ← **범위 밖.** 제안만 하고 진행하지 않음

## §3 ④단계 산출물 — 「구체안 카드」

실측을 근거로 사용자의 말을 수치로 옮긴다. 아래 5칸을 항상 채운다.

| 칸 | 내용 |
|---|---|
| **1. 수치 규격** | 폭·간격·길이를 cm와 칸 수 양쪽으로 (예: 도로 폭 2칸 = 2000cm = 20m) |
| **2. 물량** | 몇 개 놓이는지 (액터 수·조각 수). 이게 perf와 직결된다 |
| **3. perf 영향** | 베이스라인(드로우콜 920 / GPUTime 12ms) 대비 예상. Bake(ISM) 후 기준으로 |
| **4. 제약 점검** | §8 하드 제약을 어기는지 항목별로 O/X |
| **5. 선택지 + 추천** | 2~3안을 제시하고 **어느 것을 왜 추천하는지 한 줄**. 나열만 하고 끝내지 않는다 |

> 비자명한 구조 결정은 CLAUDE.md 원칙4대로 **3줄**(① 제1원리 근거 ② UE 표준과 같은가 다른가 ③ 이 프로젝트 제약과 맞는가)을 덧붙인다.

## §5 좌표 · 격자 규약 (모든 주제에서 공통으로 쓰는 말)

| 용어 | 정의 |
|---|---|
| **1칸** | 1000cm = 10m. 건물 모듈 250cm 기준으로 **정확히 4모듈** |
| **격자** | 26 × 26칸. 월드 범위 **-13000 ~ +13000** (X, Y 모두) |
| **칸 번호** | `(i, j)`, i·j는 **0~25**. 칸 `(i,j)`의 최소 모서리 = `(-13000 + i*1000, -13000 + j*1000)` |
| **바닥면** | z = 0 (평판 윗면). 모든 배치의 기준 높이 |
| **1층 높이** | 300cm. 층수 N → 건물 높이 300N |
| **방위** | +X = 동, +Y = 북 으로 부른다 (말로 소통할 때 헷갈리지 않게) |

⚠️ **26은 짝수라 원점(0,0)이 칸 모서리에 걸린다.** 중앙에 광장을 두려면 **짝수 칸 크기**(4×4, 6×6…)로 잡아야
원점 대칭이 된다. PlayerStart 4개가 원점 주위 3.5m에 있으므로 **중앙 4칸(40m)은 최소한 비워야 한다.**
→ S1에서 확정할 것.

## §6 재현성 · 되돌리기 (기술 규약)

### 6-1. 설계는 파일에 남긴다
확정된 파라미터는 대화에만 두지 않고 **스펙 파일**에 적는다.
- 위치: `Content/Python/citylayout_spec.py` (신규 예정)
- 내용: 격자 크기 · 도로 정의 · 블록 정의 · 지구별 건물 파라미터 · 랜드마크 · **시드값**
- 목적: **같은 스펙 = 같은 도시**. 다음 세션/다른 사람이 그대로 재생성할 수 있어야 한다(§9-4 검증 항목)

### 6-2. 생성물은 태그로 지운다
모든 생성 액터의 부모에 태그를 박는다. 지울 때는 태그로 찾아 지운다 — 손으로 고른 것은 건드리지 않는다.
| 태그 | 대상 |
|---|---|
| `CityGen` | `fpsr_citygen`이 만든 건물 (기존 규약) |
| `CityLayout:S1` … `S6` | 이번 작업의 단계별 생성물 (신규 규약) |
| `CityGenSize:WxDxF` · `CityGenSeed:N` | 건물 개별 재현 정보 (기존 규약) |

**단계별 되돌리기** = 해당 태그만 지우고 그 단계부터 다시 생성.

### 6-3. 손대지 않는 것 (생성·삭제 대상에서 제외)
조명 · SRS 4액터 · SkyAtmosphere · ExponentialHeightFog · PostProcessVolume · PlayerStart 4개 · 바닥 평판.
**룩이 이미 맞춰져 있어 다시 만들기 번거롭다.**

### 6-4. 에디터 · 세션 주의
- 에디터가 켜진 채로는 `git checkout`이 실패한다(`unable to unlink`). 맵을 되돌리려면 **에디터 먼저 종료**.
- **C++ 빌드가 필요해지면 세션을 쪼갠다** (세션A = 코드+빌드+커밋 / 세션B = 에디터 작업).
- Python 수정은 재빌드 불요: `import importlib, fpsr_citygen; importlib.reload(fpsr_citygen); fpsr_citygen.register_menu()`

## §9 완료 판정 (S8에서 이 5개를 전부 통과해야 U22a-B 종료)

1. **런 완주** — PIE에서 미션이 뜨고 보스까지 진행된다

## §10-1 S5 예비 메모 (소스 조사에서 미리 확보 — S5 때 나머지 축 재조사)

**보스 스폰** (`AFPSRBossSpawnPoint`)
- 에디터 노출 프로퍼티는 `Weight`(기본 1.0) · `bEnabled`(기본 true) **딱 2개**, 기본값이 이미 유효
  → 파이썬으로 그냥 스폰만 해도 동작한다. `unreal.FPSRBossSpawnPoint`.
- **1개면 충분**(N개면 가중 랜덤 풀). `bEnabled=false` 또는 `Weight<=0`이면 후보에서 제외.
- 위치와 **yaw 회전 둘 다** 쓰인다(스케일은 무시).
- 🔴 **보스는 중력이 꺼져 있고 이동이 비활성**이라 스폰 Z에 영원히 머문다. 캡슐 = 반지름 120 · 반높이 200.
  → **스폰 지점 Z = 바닥 + 200** 으로 놓아야 반쯤 묻히지 않는다(엔진 폴백 코드도 +200을 쓴다).
  → 아레나에 **지름 2.4m · 높이 4m** 여유가 필요하다.
- 스폰이 `AlwaysSpawn`이라 **건물 안에 놓아도 그대로 박혀 스폰된다**(엔진이 안 밀어낸다) → 위치 검증은 우리 몫.

**룸 점진 개방** (`AFPSRSpawnRoom` / `AFPSRDoor`)
- ✅ **완전 옵트인** — 룸이 0개여도 적 스폰은 정상 동작한다(ZoneTag 없는 스폰 지점은 항상 후보).
- 🔴 Activate 모드 룸을 놓으면 **BeginPlay에 룸 박스 안의 태그 없는 적 스폰 지점에 RoomTag를 덮어쓴다**
  → 그 지점들이 즉시 "닫힘"이 된다. 룸을 쓸 거면 이 파괴적 부작용을 알고 써야 한다.
- 룸과 문 사이에 **연결·순서 프로퍼티가 없다.** 진행 순서는 순전히 박스 배치와 플레이어 진입에서 창발한다.
- 룸당 저작 = ① EntryTrigger 박스 크기 ② 고유 `RoomTag`(미리 등록된 태그여야) ③ 딱 하나에만 `bActiveAtStart`.
  `RoomTag`가 비면 **경고 없이 완전 무동작**이 된다.
- `AFPSRDoor`는 미션/스폰 흐름과 무관하며, 단일 맵에서는 스트리밍 효과가 무의미.

> ⏳ 미조사(끊김): 미션 스폰 지점 · 런 완주 요건. **S5 착수 시 재조사할 것.**

## §10-2 맵 축소 조사에서 확정된 사실 (2026-07-22, 24에이전트 적대검증)

### 🔴 A. 맵 밖으로 떨어지면 게임이 끝난다 — 맵 크기와 무관한 기존 결함
| | |
|---|---|
| KillZ | **엔진 기본 −1,048,575cm**. `Config/DefaultEngine.ini` 오버라이드 0건, 맵에 `KillZVolume` 0개 |
| 낙하 시간 | 중력 980 · 종단속도 4000cm/s → **약 4분 24초** 낙하 후 `FellOutOfWorld` → `Destroy()` |
| 부활 | **없다.** `AFPSRGameMode`가 오버라이드하는 건 `BeginPlay`·`Logout` 둘뿐이고 `RestartPlayer`를 부르지 않는다 |
| 적은? | 적은 자동 회수된다(`WorldKillZ` 검사, 약 3.9초 — 30fps·거리LOD 최악 6.4초). **적용 벽은 불필요** |

→ **필요한 조치 2건**(경계 벽을 어떤 방식으로 만들든 별개로 필요):
① `Map_CyberCity` WorldSettings의 **KillZ를 −3000 정도로** 올려 3초 안에 판정되게 한다
② `AFPSRCharacter::FellOutOfWorld`를 오버라이드해 Destroy 대신 **즉사(기존 DBNO 경로) 또는 마지막 유효 위치로 텔레포트**

### B. 경계 벽 규격 (플로우필드가 실제로 인식하는 조건)
- 베이크 프로브 = **반크기 (40,40,60)cm 박스 2종**(셀 중심 / 셀 경계 중점), 100cm 간격
  → **두께 20cm 미만 벽은 프로브 사이로 새어나간다.** 하한 40cm, **목표는 200cm(=1셀)**
  1셀 두께면 정렬과 무관하게 항상 잡히고 벽 안쪽 셀이 통째로 blocked가 돼 적이 벽에 비비지 않는다
- 프로브 수직 창 = 바닥+60 ~ +180cm → 60cm 미만 턱은 애초에 안 보인다(45/60 규칙과 정합)
- **적 스텝업 하드 상한 = 180cm**(`GroundSnapTolerance` 60 × 3회, `MaxCrestStepUp=180`). 벽은 이보다 훨씬 높게

### 🔴 C. "스폰 거리 상한 80m"는 **이 맵에 존재하지 않는다** (이전 리포트 S4-1의 오류 정정)
`TrySelectSpawnPoint`(`FPSREnemySpawnSubsystem.cpp:1209-1281`)의 후보 필터는 **네 개뿐**:
MapId 일치 / `IsEnabled()` / 활성 룸 존 태그 / `MinPlayerDistance`(**기본값 0.0** = 꺼짐).
통과한 후보 중 **균등 랜덤**(거리 가중치는 2026-06-25 제거됨).
`ChaseEnterCells=40`(80m)은 `ComputeFrontState` 전용이고, 그 함수는 멀티슬롯 통합 컴퓨터가 없으면
즉시 반환한다 — **단일 맵인 Map_CyberCity에서는 한 번도 실행되지 않는다.**
→ **적이 어디서 나오는지는 100% 스폰 포인트를 어디에 찍느냐로만 정해진다**(시야 게이트도 이미 제거됨).

### D. 밀도 이득은 1.8배지 3배가 아니다 (내 보고 정정)
도로 면적은 `half`에 대해 **선형**으로 줄고 판 면적은 **제곱**으로 준다.
| | 판 면적 | 도로·인도·광장 | 적 1마리당 |
|---|---|---|---|
| 260m | 67,600 m² | 17,000 m² | 88.5 m² |
| **150m** | 22,500 m² | **9,300 m²** | **48.4 m²** |
또한 **264m → 200m 축소는 밀도 이득이 정확히 0**이다(드레인 마름모가 200m부터 상수). 곡선은 200m 아래부터 움직인다.

### E. 시야 실측 (150m 레이아웃)
광장 중앙에서 5° 간격 레이캐스트: 큰길 축(0°/90°)은 **맵 끝까지 뚫림**, 45° 대각선은 **14.3m에서 차단**
(모서리 건물). 90° 부채꼴 중 약 50°가 30m 안에서 차단된다. 사분면 내부 빈 공터 대각선 = **81m**.
