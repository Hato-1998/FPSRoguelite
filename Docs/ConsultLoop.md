# ConsultLoop — 게임 컨설턴트 토론 프로토콜 (백엔드 Claude × 클라이언트 Codex)

> **이 문서는 "컨설턴트 시스템"의 정형화된 프로토콜이다.** 사용자가 *"`ConsultLoop.md` 읽고 `<주제>` 토론 결과 줘"* 또는 `/consult <주제>` 라고 하면, Claude는 이 문서의 절차대로 **백엔드 렌즈(자신) ↔ 클라이언트 렌즈(Codex)** 를 왕복 토론시켜 수렴한 뒤 결과를 보고한다.
> 작업 시작 전 `Game.md`(SSOT 허브)·`PROGRESS.md`·주제 관련 `Docs/SSOT/*.md`를 먼저 읽는다(허브 §0-1 라우팅).

---

## §0. 목적 · 위치 (무엇이며, 무엇이 아닌가)

프로젝트를 **두 전문가 렌즈로 동시에** 검토받기 위한 라이브 토론 채널이다. 한쪽 시야의 사각(서버는 깔끔한데 손맛이 죽거나, 손맛만 보다 복제·성능 예산을 깨는 등)을 교차로 메운다.

기존 외부 AI 협업과 **역할이 다르다 — 제3 채널**:

| 채널 | 트리거 | 용도 | 산출물 |
|---|---|---|---|
| `Scripts/codex-review.ps1` (`codex review`) | 단계 완료·커밋 직전 | 코드 **diff 게이트**(원칙 위반·버그) | `Docs/codex-reviews/` (gitignore) |
| `GameConfirm.md` | 비정기 | 외부 AI가 **문서를 리뷰**(외부가 작성) | 외부 |
| **ConsultLoop** (`Scripts/consult-codex.ps1`, `codex exec`) | 사용자가 주제 지목 | 백엔드×클라 **설계/콘텐츠 라이브 토론(컨설팅)** | `Docs/Review/` (**추적**, 프롬프트 매니저 인입원) |

→ ConsultLoop은 **자문(advisory)** 이다. 토론 자체로는 코드/에셋을 바꾸지 않는다(§7).

## §1. 역할 정의 (두 렌즈)

| | **🔧 백엔드/시스템 렌즈 = Claude** | **🎮 클라이언트/콘텐츠 렌즈 = Codex** |
|---|---|---|
| 1차 관심 | 서버 권위·복제(Push Model)·GAS 경계·성능 예산(§5)·데이터드리븐 C++ 베이스·확장성·네트워크 정합 | 손맛/게임필·입력 반응성·UI/HUD·애님/VFX/사운드 코스메틱·플레이어 경험·콘텐츠 데이터 구성 |
| 묻는 질문 | "이게 수백 적·4인 협동·서버권위에서 비용/복제가 견디나? 다음 단계로 확장되나?" | "플레이어가 이걸 어떻게 *느끼나*? 입력→피드백 루프가 즉각적인가? 콘텐츠로 표현 가능한가?" |
| 산출 | 시스템 입장·리스크·예산 영향 | 체감 입장·연출/피드백 요구·콘텐츠 실현성 |

**공통 구속 (둘 다 반드시 지킴):**
- **장르 정체성(`Game.md` §1)** — 1인칭 FPS × 뱀파이어 서바이벌 × 4인 협동 로그라이트. **제1원리 = 적 수백(동시 ~200-300 실측) 싸게**(표준 Hero Shooter/Lyra 스택 그대로 아님). 어느 렌즈도 이 정체성을 깨는 제안은 무효.
- **제1원리 우선, 레퍼런스 맹종 금지(`CLAUDE.md` 4원칙)** — Lyra는 *플레이어 측* 한정 교차검증 레퍼런스. "Lyra가 그러니까"는 근거가 아님.
- Codex는 리포 **`AGENTS.md`를 자동 로드**해 위 구속을 이미 인지한다. 프롬프트엔 역할 프라이머만 덧붙인다(§4).

## §2. 트리거

- 자연어: **"`ConsultLoop.md` 읽고 `<주제>` 토론 결과 줘"**
- 슬래시: **`/consult <주제>`** (`.claude/commands/consult.md`)
- 주제는 임의 — 구조(예: "스폰 디렉터 복제 모델", "카드 모디파이어 적용 순서")든 콘텐츠(예: "스핀업 LMG 손맛 튜닝 방향", "보스 페이즈 연출")든.
- 사용자가 *"시간이 얼마나 걸리든"* 이라 했으면 **수렴까지** 돈다(§3). 라운드 수 하드 캡 없음.

---

## §3. 토론 프로토콜 (수렴 기반)

**R0 — 범위 설정 (Claude 단독, 호출 전):**
1. 주제 관련 `Docs/SSOT/*.md` + 관련 소스/콘텐츠를 읽어 **사실 기반**을 잡는다(추측 금지).
2. 의제를 2~4개 핵심 쟁점으로 프레이밍한다.

**R1 — 입장 제시:**
1. Claude가 **백엔드 렌즈 입장**을 작성(쟁점별 시스템 관점 + 리스크/예산).
2. 그 입장 + 의제 + 클라 렌즈에 묻는 구체 질문을 프롬프트 파일에 적고 → `Scripts/consult-codex.ps1 -PromptFile <file>` 로 **Codex의 클라 렌즈 입장**을 받는다(§4).

**R2+ — 교차 반론 (반복):**
1. Claude가 Codex 출력을 **판독**(동의/이견/오해 구분). 장르·예산 위반 주장은 §1 근거로 반박.
2. 조정된 백엔드 입장 + 남은 질문을 다시 보내 Codex 응답을 받는다.
3. **새로운 실질 논점이 더 안 나오면 수렴**으로 보고 종료(통상 3~6라운드, 단순 주제는 1~2, 복잡하면 더).

**수렴 정리 — 세 갈래로 분류:**
- **(a) 합의** — 양 렌즈가 동의한 권고.
- **(b) 미해결 쟁점** — trade-off가 남아 결정 못 한 것(양측 근거 병기).
- **(c) 사용자 결정 필요** — 취향·범위·우선순위 등 사람이 정해야 하는 것.

## §4. Codex 호출법

```powershell
# 한 라운드: 프롬프트 파일 작성 → 호출
Scripts\consult-codex.ps1 -PromptFile <round.txt> -Title <주제슬러그>
# 빠른 단발/검증:
Scripts\consult-codex.ps1 -Prompt "<짧은 질문>" -DryRun   # 호출 없이 명령/프롬프트만 확인
```

- 래퍼가 `codex -C <repo> exec -s read-only -` (read-only 샌드박스, 프롬프트 stdin)로 호출. Codex가 `AGENTS.md`+리포를 읽어 답한다.
- **프롬프트 파일 권장 구조** (Claude가 매 라운드 작성):
  1. *역할 프라이머* — "너는 이 프로젝트의 **클라이언트/콘텐츠 기술 컨설턴트**다. 손맛·입력·UI·연출·콘텐츠 실현성 관점에서 답하라. 장르 정체성(Game.md §1, Hero Shooter 아님)과 제1원리를 지켜라."
  2. *주제 + 현재까지의 의제/합의/쟁점 요약*
  3. *백엔드 렌즈의 이번 라운드 입장*
  4. *Codex에게 묻는 구체 질문 2~4개* (열린 토론이 아니라 **쟁점에 답하게**)
- Codex 미설치/미인증이면 래퍼가 안내 후 멈춘다 → `npm install -g @openai/codex`; `codex login` 1회.
- codex 경로 강제: `$env:CODEX_EXE = '<경로>'`.

## §5. 산출물 포맷

`Docs/Review/<yyyyMMdd>-<주제슬러그>.md` 에 저장(**추적** — 설계 의사결정 기록 + 프롬프트 매니저 인입원):

```markdown
# 컨설트: <주제> (<날짜>)
## 범위 / 읽은 컨텍스트      ← 어떤 SSOT/소스를 근거로 했나
## 🔧 백엔드 렌즈 핵심 입장 (Claude)
## 🎮 클라이언트 렌즈 핵심 입장 (Codex)
## 토론 로그 요약           ← 라운드별 핵심 교환 (장황한 원문 X)
## ✅ 합의 권고
## ⚖️ 미해결 쟁점           ← trade-off + 양측 근거
## 🙋 사용자 결정 필요
## 📌 액션 아이템           ← 갱신할 Docs/SSOT 파일 / 반영할 로드맵 유닛
```

- 사용자에겐 채팅으로 **압축 요약**(합의/쟁점/결정필요/액션)을 보고하고, 전체는 위 파일을 가리킨다.
- **프롬프트 매니저 인입**: 리포트 저장 후 `📌 액션 아이템`을 `Docs/TaskPrompts_Master.md` §E 인입 표에 한 줄로 등재(주제·리포트 링크·대상 유닛). 프롬프트 매니저가 이를 읽어 백로그/유닛에 반영한다.
- 원시 Codex 응답은 `Docs/Review/_raw/`에 타임스탬프로 남는다(gitignore, 감사용).

## §6. 가드레일

1. **자문 전용** — 토론은 코드/에셋을 바꾸지 않는다. 채택된 설계는 **별도 단계**에서 해당 `Docs/SSOT/` 도메인 파일을 먼저 갱신(원칙3) 후 구현.
2. **사실 기반** — R0에서 실제 SSOT/소스를 읽는다. 두 렌즈 모두 추측으로 단정 금지(엔진 API는 §6-3대로 소스 대조).
3. **장르·제1원리 우선(§1)** — 어느 렌즈의 제안이든 §1 구속을 깨면 무효. 비자명한 구조 결론은 ①제1원리 ②(참고)Lyra/UE표준 비교 ③프로젝트 제약 정합 3줄로 명시.
4. **수렴이 목표, 합의 강요 아님** — 안 좁혀지는 trade-off는 **(b)쟁점/(c)사용자결정**으로 정직하게 남긴다. 억지 단일 결론 금지.
5. **read-only** — Codex는 read-only 샌드박스로 호출(쓰기 권한 없음). 신뢰 로컬 리포 전용.
